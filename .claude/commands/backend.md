백엔드 문서화 에이전트입니다. 현재 열려 있는 Pencil 디자인 파일을 읽고, 데이터 모델·API 엔드포인트·비즈니스 로직·유효성 검사 규칙을 기능별로 정리한 마크다운 문서를 생성합니다.

## 단계별 작업

### 1단계 — 디자인 파일 읽기

현재 에디터 상태를 가져와 전체 디자인을 파악합니다:
- `mcp__pencil__get_editor_state` — 모든 페이지와 문서 구조 확인
- `mcp__pencil__get_guidelines` — 주석된 스펙 수집
- `mcp__pencil__get_variables` — 명명된 데이터 필드와 상태 값 추출
- `mcp__pencil__snapshot_layout` — 각 페이지의 전체 요소 트리 읽기
- `mcp__pencil__batch_get` — 텍스트 레이블, 폼 필드, 테이블 컬럼, 리스트 항목, 상태 값 읽기 (데이터 구조 파악에 핵심)
- `mcp__pencil__get_screenshot` — 페이지를 시각적으로 확인하여 흐름 이해

`$ARGUMENTS` 값이 있으면 해당 이름과 일치하는 기능만 문서화합니다. 없으면 모든 페이지를 분석해 논리적인 백엔드 기능 단위로 묶습니다.

### 2단계 — UI에서 기능 도출

UI를 보고 다음 패턴으로 백엔드 기능을 추론합니다:

- **목록/테이블 화면** → CRUD API가 필요한 엔티티
- **폼 화면** → 생성/수정 뮤테이션, 필드 유효성 검사 규칙
- **상세 화면** → 조회 API, 중첩 데이터 관계
- **필터/검색 UI** → 쿼리 파라미터, 검색 인덱싱 필요 여부
- **인증 화면** (로그인, 회원가입, 프로필) → 인증 기능
- **상태 배지, 진행률 표시** → 상태 머신(State Machine), 열거형(Enum)
- **알림 UI** → 알림 또는 이벤트 시스템
- **설정 화면** → 사용자 환경설정, 설정 API
- **업로드/미디어 UI** → 파일 저장소 기능
- **차트/분석 화면** → 집계 쿼리, 리포팅 기능

관련된 페이지들을 하나의 기능으로 묶습니다. 예: "상품 목록", "상품 상세", "상품 추가" 페이지는 모두 "상품 관리" 기능으로 통합합니다.

### 3단계 — 기능별 마크다운 파일 생성

각 기능별로 `docs/backend/{기능명}.md` 파일을 아래 구조로 생성합니다:

```markdown
# {기능 이름}

> {이 기능이 무엇을 가능하게 하는지 한 줄로 설명}

**관련 UI 페이지**: {이 기능과 연결된 페이지 목록}

---

## 데이터 모델

### {엔티티명}

| 필드 | 타입 | 필수 여부 | 설명 |
|---|---|---|---|
| `id` | UUID / number | 필수 | 기본 키 |
| `{필드}` | string / number / boolean / Date / enum | 필수 / 선택 | {UI에서 추론한 설명} |

**열거형(Enum) 값** (UI 레이블/배지에서 추론):
```
{EnumName}: "값_A" | "값_B" | "값_C"
```

**관계**:
- `{엔티티}`는 `{다른엔티티}`에 속함 (다대일)
- `{엔티티}`는 `{다른엔티티}` 여러 개를 가짐

## API 엔드포인트

### {간단한 설명}
```
{메서드} /api/{리소스}
```

**인증 필요**: 예 / 아니오

**쿼리 파라미터** (목록/검색 엔드포인트):

| 파라미터 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `page` | number | 아니오 | 페이지네이션 |
| `limit` | number | 아니오 | 페이지당 항목 수 |
| `{필터}` | string | 아니오 | UI 필터 칩에서 추론한 필터 |

**요청 본문** (POST/PUT/PATCH):
```json
{
  "{필드}": "{타입} — {설명}"
}
```

**응답** (성공):
```json
{
  "data": "{UI에서 추론한 데이터 구조}",
  "meta": {
    "total": "number",
    "page": "number"
  }
}
```

**에러 응답**:

| 코드 | 상황 |
|---|---|
| 400 | 유효성 검사 오류 |
| 401 | 미인증 |
| 403 | 권한 없음 |
| 404 | 리소스 없음 |

{각 엔드포인트에 대해 반복}

### 4단계 — 인덱스 업데이트

모든 기능 문서 생성 후 `docs/backend/INDEX.md`를 업데이트합니다:

```markdown
# 백엔드 문서 인덱스

Pencil 디자인 파일 기준 생성일: {날짜}

| 기능 | 파일 | 관련 페이지 | 주요 엔티티 |
|---|---|---|---|
| {기능 이름} | [{기능명}.md]({기능명}.md) | {페이지들} | {엔티티들} |
```

### 품질 원칙
- 디자인에서 보이는 것만 기반으로 추론합니다 — 임의로 요구사항을 만들지 않습니다
- UI에서 확인할 수 없는 내용은 `<!-- TODO -->`로 표시합니다
- 필드 타입은 TypeScript 표기법을 사용합니다: `string`, `number`, `boolean`, `Date`, `string[]`, 또는 특정 열거형
- 디자인에서 달리 암시되지 않으면 REST 관례를 따릅니다
- 같은 엔티티가 여러 페이지에 나타나면 모델을 한 번만 정의하고 참조합니다
- 복잡한 상태 머신이 암시되는 UI 상태는 강조 표시합니다 — 백엔드 설계에 주의가 필요합니다
